# FAQ 与排障

这份 FAQ 只覆盖当前仓库里的真实实现，不讨论已经移除或尚未实现的方案。

## 为什么必须配置主站

主站负责写入主数据骨架，包括：

- 检索索引
- 详情信息
- 全局影片归算

附属站主要补充播放列表。没有主站时，附属站数据无法稳定挂载到完整影片上，前台搜索和详情也缺少基础数据。

## 主从采集现在怎么分工

- 主站：负责影片主数据
- 附属站：负责补充播放源
- 后台支持对同一影片执行“更新全部站点”
- 两者会被归并到同一部影片下

这也是为什么切换主站后，系统仍需要重新建立主数据骨架。

## 为什么会有内容去重

当前实现会优先使用豆瓣 ID；缺失时再退化到标题等信息生成内容指纹。这样可以把不同采集源的同一部影片归并到同一条主记录下，避免重复详情和重复搜索结果。

## 为什么修改配置后前台没有立刻变化

常见原因有三类：

- 浏览器缓存
- CDN / 反向代理缓存
- 你修改的是未公开给前台使用的字段

后端针对首页、筛选标签、TVBox 配置等缓存已经有主动清理逻辑，但浏览器层缓存仍然可能让你看到旧结果。

补充说明：

- 服务启动时只会清理项目自身缓存
- 主站切换、分类重建后也会主动刷新相关结果
- 如果你看到旧的排序项或旧的筛选项，优先怀疑当前运行实例没有重启到最新代码，或旧缓存还没有刷新

## 现在“最近更新”到底看什么

当前实现已经收敛为单一语义：

- 最近更新只看主站资源更新时间

这意味着：

- 主站数据变化会影响最近更新
- 附属播放源同步不会改变最近更新顺序
- 公共分类搜索和 TVBox 列表对最近更新的理解是一致的

如果是旧部署升级上来的环境，历史残留数据仍可能需要额外清理。

## 为什么公共分类页和 TVBox 结果现在应该一致

因为当前实现已经把两边统一到同一套查询逻辑：

- 相同的分类条件
- 相同的 `剧情 / 地区 / 语言 / 年份` 筛选解释
- 相同的 `其他` 处理逻辑
- 相同的排序语义

如果两边结果仍不一致，优先检查缓存和运行实例，而不是先怀疑前端页面本身。

## 为什么后台页面能打开，但接口又提示未登录

这是当前设计决定的：

- 前端 `/manage` middleware 只检查登录 cookie 是否存在
- 后端 `/api/manage/*` 才做真正的 JWT 与 Redis 校验

所以页面“能进壳子”不代表接口一定有效，最终以后台接口返回结果为准。

## 为什么 Docker 往往还要配置 `API_URL`

因为：

- 浏览器端会先请求当前站点下的 `/api/*`
- 服务端渲染请求也需要一套明确的后端地址

现在前端会显式使用 `API_URL`：目录内单独运行 `web` 时来自 `web/.env.local`，Compose 运行时来自 `docker-compose.yml` 注入。

但 Docker 场景不同：容器里的 `127.0.0.1` 指向的是 `web` 容器自身，不是 `server` 容器，所以仍然需要显式写成类似 `http://server:8080` 的容器内地址。

## 为什么 `web` 和 `server` 会端口冲突

如果你本地单独运行 `web` 时又手动把 Next 开发端口改成了 `8080`，同时 `server/.env` 里后端也监听 `8080`，就会产生冲突。

当前仓库的约定是：

- `server/.env` 中的 `PORT` 只给后端 API 使用
- `web` 本地开发默认使用 Next 的 `3000`
- Compose 部署时固定暴露 `web:3000` 和 `server:8080`

## 为什么登录态用 cookie，而不是 localStorage

当前后台鉴权已经统一到 `HttpOnly` cookie：

- 更适合服务端控制登录态
- 能减少前端自己维护 token 的复杂度
- 更符合当前 Next + Go 分离项目的实际使用方式

## 为什么 Docker 环境连不上宿主机数据库

优先检查这些点：

- `MYSQL_HOST` 是否写成了容器可访问地址
- Linux 环境下 `host.docker.internal` 是否可用
- 数据库用户是否允许从容器来源 IP 连接
- 宿主机防火墙是否拦截

## 默认账号有哪些

文档仅公开内置访客账号：

- 访客只读：`guest / guest`

访客账号可以正常查看后台页面和列表接口，但所有写操作都会被后端直接拦住。管理员账号不在公开文档中暴露，这些内置账号仅适合初始化或演示环境，不适合长期保留默认密码。

## 还有哪些已知注意项

- `/api/config/basic` 目前仍是公开接口
- 前端 `eslint` 仍有图片相关 warning
- 当前 Compose 已包含可选的 MySQL / Redis 服务；是否启动取决于你的启动命令与 `docker-compose.yml` 中的配置
