# FAQ 与排障

这份文档只覆盖当前仓库的真实实现。

## 采集与数据

### 为什么必须配置主站

主站负责保存影片的基础信息、分类、详情和前台列表。没有主站时，附属站的播放地址没有地方挂载，前台搜索和详情页也没有基础数据可用。

### 主站和附属站怎么分工

- 主站写入影片主数据。
- 附属站写入播放列表。
- 详情页和播放页会把附属站播放源聚合到主站影片下。
- 后台支持对同一影片执行“更新全部站点”。

### 为什么切换主站后要重建数据

主站决定影片基础信息、分类和搜索结果。切换主站后，需要用新主站重新整理数据，避免新旧站点的分类、详情和列表混在一起。

### 为什么会做内容去重

系统会优先用豆瓣 ID 判断是不是同一部影片；没有豆瓣 ID 时，再用标题等信息辅助判断。这样可以减少重复影片和重复搜索结果。

### 手动停止采集会发生什么

停止会阻止继续采集新的分页，并取消仍在运行的任务。已经成功入库的数据会继续完成整理，尽量保证这些内容能被前台读到。

如果停止前已经有部分页面写入成功，前台看到的可能是这次停止前已成功入库的部分增量。

### 为什么搭建完成看不到影片

EcoHub 不内置影视数据。首次搭建完成后，需要先在后台配置采集源，并执行采集任务。

采集任务完成后，系统还会整理列表和筛选数据。整理完成后，前台首页、分类、搜索、详情页、后台影片列表以及 TVBox / MacCMS 接口才能看到这些影片。

## 快照、筛选与缓存

### 为什么采集完成后还要发布快照

采集过程中，数据会不断变化。系统会在采集结束后统一整理一份给页面使用的数据，前台列表、筛选和 TVBox 接口读取的是这份整理后的结果。这样页面更稳定，访问速度也更可控。

### 增量发布为什么仍然可能耗时

即使只更新一部分影片，系统仍然需要完成这些工作：

- 检查影片和详情是否有效。
- 更新前台列表数据。
- 更新分类、年份、地区等筛选数据。
- 更新服务运行中的缓存。

一次更新的影片很多时，数据库写入和缓存刷新仍然会占用时间。

### 为什么修改配置后前台没有立刻变化

常见原因：

- 浏览器缓存。
- CDN 或反向代理缓存。
- 当前运行实例不是最新代码。
- 修改的是前台不会展示的字段。
- 采集后的列表和筛选数据还没有整理完成。

后端会在主站切换、分类重建、列表发布后刷新项目自身缓存，但无法控制浏览器、CDN 或外部代理缓存。

### 为什么公共分类页和 TVBox 结果应该一致

两者使用同一套查询规则：

- 相同分类条件。
- 相同剧情、地区、语言、年份筛选解释。
- 相同“其他”处理逻辑。
- 相同排序规则。

如果仍不一致，优先检查缓存、运行实例和请求参数。

### “最近更新”到底看什么

最近更新只看主站资源更新时间。

这意味着：

- 主站数据变化会影响最近更新。
- 附属播放源同步不会改变最近更新排序。
- 前台分类页和 TVBox 列表排序语义一致。

## 登录与权限

### 为什么后台页面能打开，但接口提示未登录

`/manage` 页面只由前端做 cookie 存在性预检查。真正的 JWT 校验、Redis token 校验和权限判断都在后端 `/api/manage/*`。

页面能进入壳子不代表接口一定有效，最终以后端接口返回为准。

### 为什么使用 cookie 而不是 localStorage

后台登录态使用 `HttpOnly` cookie：

- 前端不需要自己保存 token。
- 后端统一控制登录态和自动续期。
- 减少 token 被前端脚本读取的风险。

### 访客账号能做什么

访客账号可以查看后台数据，但所有写操作会被后端 `WriteAccess` 中间件拒绝。

演示访客账号：`guest / guest`。

### 默认账号是否可以直接用于生产

不建议。默认账号只适合初始化和演示。正式部署后应立即修改密码，或替换为自己的账号体系。

## Docker 与部署

### Docker 里的 `API_URL` 需要手动配置吗

不需要手动配置。Compose 会按 `SERVER_PORT` 自动生成：

```env
API_URL=http://server:${SERVER_PORT:-8080}
```

`API_URL` 是前端容器访问后端容器时用的内部地址。Docker 容器里的 `127.0.0.1` 指向容器自己，不是后端容器，所以不要在根目录 `.env` 中配置它。

需要注意的是，`/api` 不是前端独有前缀。后端接口地址本身也以 `/api` 开头，所以通过 `SERVER_PUBLIC_PORT` 直连后端时同样要访问 `/api/health`、`/api/provide/config` 这类路径。

### Docker 环境连不上宿主机数据库怎么办

优先检查：

- `MYSQL_HOST` / `REDIS_HOST` 是否是容器可访问地址。
- Linux 环境下 `host.docker.internal` 是否可用。
- 数据库用户是否允许容器网络来源连接。
- 宿主机防火墙是否放通端口。
- MySQL / Redis 是否只监听了 `127.0.0.1`。

### 使用内置 MySQL / Redis 启动失败怎么办

优先检查：

- 宿主机 `3306` 或 `6379` 是否已被占用。
- `docker compose logs -f mysql`。
- `docker compose logs -f redis`。
- `server.environment` 中的数据库、Redis 密码是否和服务配置一致。
- 内置 MySQL / Redis 默认不对宿主机暴露端口，不会占用宿主机 `3306` / `6379`。

### 前端打开后接口全部失败怎么办

优先检查：

- `server` 容器是否健康。
- `web` 容器中的 `API_URL` 是否与 `SERVER_PORT` 一致。
- 反向代理是否正确转发 `/api/*`。
- 浏览器请求是否被 HTTPS、CORS 或 cookie 策略拦截。

### `web` 和 `server` 为什么会端口冲突

本地开发约定：

- `server` 默认监听 `8080`。
- `web` 默认监听 `3000`。

如果手动把 Next 改到 `8080`，就会和后端冲突。修改 `web/.env.local` 中的 `PORT` 后需要重启前端服务。

## 已知注意项

- `/api/config/basic` 当前仍是公开接口。
- 前端 lint 可能仍有图片优化相关 warning。
- 是否启动内置 MySQL / Redis，取决于你执行的 `docker compose up` 命令里包含哪些服务。
- 一次整理特别多影片时，服务会有短暂的内存上涨。
