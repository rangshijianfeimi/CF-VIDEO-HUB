# 采集链路重构说明

## 重构立场

本次重构严格遵守以下原则：

1. 不做向后兼容
2. 允许直接调整表结构与字段语义
3. 不保留旧链路作为兜底
4. 不做过渡态双写
5. 不做历史数据平滑迁移

这不是一次“局部优化”，而是对现有采集、分类、展示链路的直接替换。

旧设计只要与新目标冲突，就直接删除、改写或废弃，不保留兼容入口。

## 问题定义

当前系统的核心问题不是实现细节，而是模型本身错误。

现有链路把以下几类数据混在一起处理：

- 采集事实数据
- 分类规则数据
- 分类投影数据
- 搜索展示数据
- 缓存与标签数据

直接后果如下：

1. 修改分类规则会回写采集资源数据
2. 主站采集结果依赖后置重建才能按当前规则稳定展示
3. 分类重建会串联搜索标签重建
4. 系统长期依赖“事后修正资源数据”维持一致性
5. 数据量一大，规则修改就会触发高成本全量重算

这套设计必须整体废弃，不能继续在上面做修补。

## 重构目标

重构后的系统必须满足以下硬约束：

1. 主站采集只保存资源站原始分类身份和资源事实，不固化会随规则变化的最终分类归属
2. 分类规则定义当前展示视图，所有已采集资源不分新旧都按当前规则展示
3. 后续修改分类或规则不得回写资源事实数据，只能刷新分类映射与缓存
4. 附属站只负责播放源补充，不参与分类与展示结果计算
5. 切换主站时，继续清空旧主站数据并重新同步新主站分类

一句话定义新模型：

**采集保存来源事实，规则驱动当前展示视图。所有资源不分新旧都按当前规则展示，附属站只补播放源。**

## 新模型

### 主站

主站是唯一的数据框架来源，负责：

1. 写入影片详情事实
2. 写入来源分类事实
3. 写入来源一级/二级分类稳定身份
4. 写入不依赖规则变化的基础展示数据

主站采集完成后，资源事实即可使用；最终分类归属由当前分类规则和映射在查询时解释，不再依赖后置回写修正资源数据。

### 附属站

附属站只负责：

1. 写入播放列表
2. 补充来源映射
3. 刷新播放源相关摘要

附属站禁止负责：

1. 分类匹配
2. 分类树更新
3. 分类映射重建
4. `search_info` 分类字段修正
5. 标签重建

### 规则生效机制

分类规则和分类树只在主站采集时生效。
分类规则和分类树定义当前展示视图。

行为约束必须固定为：

1. 采集时写入来源分类稳定身份，例如 `root_category_key` / `category_key`
2. 分类规则维护“来源分类身份 -> 当前展示分类”的映射
3. 前台点击展示分类时，通过当前映射展开到来源分类身份集合后查询资源
4. 后续规则修改立即影响当前展示视图，但不得回写资源事实数据
5. 新规则生效不需要重新采集；只需要刷新映射与相关缓存

这里不允许保留“规则修改后自动回写资源分类字段”的旧逻辑。

## 数据职责重定义

### 事实层

只保存采集事实，不承担回写修正责任。

包括：

- `movie_detail_info`
- `source_categories`

### 规则层

只保存当前规则，不负责回写资源事实。

包括：

- `mapping_rules`

### 展示层

保存资源基础展示字段和来源分类身份；最终分类视图由当前规则映射解释。

包括：

- `search_info`

`search_info` 的新语义必须明确为：

**资源展示事实表，不再保存会随规则变化而反复回写的最终分类归属。**

### 分类框架层

表达当前主站下的分类框架和当前展示映射，不承担资源事实修正责任。

包括：

- `categories`
- `category_mappings`

### 缓存层

只做缓存或附加投影，不反向驱动主链路。

包括：

- `search_tag_items`

## 强制删除的旧行为

以下行为与新模型冲突，必须直接删除，不保留兼容分支：

1. 规则增删改后自动触发 `CategoryRebuild`
2. 分类重建后批量回写资源 `search_info` 分类字段
3. 为修正规则结果而全量重建搜索标签
4. 规则重载时顺带刷新分类缓存的混杂行为
5. 将 `search_info` 作为可被规则反复修正分类归属的中间表使用
6. 将当前展示分类结果依赖到事后异步回写任务

这些逻辑不是“可优化项”，而是必须移除的错误设计。

## 数据结构调整

本次重构允许直接改表，不考虑旧数据兼容。

### 必改项

`search_info` 增加：

- `category_version`
- `rule_version`

用途：

1. 标记该条资源写入时对应的来源分类框架版本
2. 标记该条资源写入时的规则版本，仅作为排障和追踪依据
3. 不得用于判定“新旧数据”或决定是否展示

### 可直接同步改造项

`movie_detail_info` 可同步增加：

- `category_version`
- `rule_version`

如果主站事实与展示事实都要保留同一轮采集快照标记，就直接一起加，不做犹豫。

如果后续发现 `categories`、`category_mappings`、`search_info` 的字段定义不再适合新模型，也可以直接改字段名、删字段或重建表结构，不保留旧语义。

### 分类身份

分类身份必须绑定主站来源分类，而不是绑定本地自增 ID 或分类名称路径。

固定语义如下：

1. `categories.id` 只作为后台 UI 与关系行 ID 使用，可随分类重建变化
2. `categories.stable_key` 使用 `source:<source_id>:<source_type_id>`
3. `search_info.root_category_key` 写入来源根分类 Key
4. `search_info.category_key` 写入来源子分类 Key
5. 前台继续接收当前分类 ID，但查询时必须先解析到 `stable_key` 再查 `root_category_key` / `category_key`

因此，分类显示、隐藏、排序、展示名称和合并/拆分规则修改都不需要重新采集；影片入库时只锁定来源分类身份。规则变更通过当前 `category_mappings` 改变展示视图，所有资源不分新旧都按当前映射展示，但不得通过后台分类保存隐式回写资源分类字段。

## 新写入流程

### 主站采集写入流程

主站采集流程应收敛为：

1. 写入 `movie_detail_info`
2. 解析来源分类
3. 写入来源根分类与子分类稳定身份
4. 生成基础展示字段
5. 直接写入 `search_info`
6. 刷新该影片自己的标签或相关最小影响面缓存

这里不允许再存在：

1. 先写半成品，再等待异步重建修正资源分类字段
2. 采集完成后再跑全量分类回写
3. 通过规则修改去回补资源展示事实

### 附属站采集写入流程

附属站采集流程应收敛为：

1. 写入 `movie_playlist`
2. 维护来源映射
3. 刷新对应影片的播放摘要

附属站不得写入或触发：

- 分类树
- 分类映射
- `search_info` 分类结果
- 历史分类修正
- 全量标签重建

## 主站切换行为

主站切换逻辑保留，但语义更明确：

1. 主站提升、降级、URI 变化时，直接清空旧主站全量数据
2. 清空后直接同步新主站分类树
3. 新主站重新全量采集，生成新的事实数据与展示结果

这里不做任何兼容迁移，也不保留旧主站数据的过渡逻辑。

## 实施策略

### 第一阶段：切断旧回写链路

直接删除以下触发关系：

1. 规则变更 -> 分类全量重建
2. 分类重建 -> `search_info` 分类字段回写
3. 规则变更 -> 全量搜索标签重建

目标：规则修改只更新当前展示映射并失效相关缓存，不回写资源数据。

### 第二阶段：主站采集直接固化来源分类身份

直接把来源分类身份写入主站采集路径，确保资源可被当前规则映射查询。

目标：主站采集结束时，`search_info` 已经包含稳定来源分类身份和基础展示事实。

### 第三阶段：收紧附属站职责

删除附属站对分类链路的一切影响。

目标：附属站只保留播放源补充职责。

### 第四阶段：清理旧模型残留

删除所有仍依赖“事后回写资源数据”的旧函数、旧任务、旧字段语义。

目标：代码里只保留新模型，不保留双轨并存。

## 验收标准

重构完成后，必须满足以下结果：

1. 主站采集完成后，展示数据可直接使用
2. 修改分类规则后，所有资源不分新旧都按当前规则展示
3. 新规则不需要重新采集即可生效，但只能通过映射和缓存影响展示视图
4. 附属站采集不会触发分类重算
5. 附属站采集不会改写主站展示分类结果
6. 切换主站时，旧主站数据仍会被直接清空
7. 系统不再依赖任何“规则修改后回写资源数据”的后台任务

## 最终结论

本次重构不是为旧链路补洞，而是直接换模型。

新的系统约束只有四条：

1. 主站决定框架
2. 主站采集时直接产出来源分类身份和展示事实
3. 附属站只补播放源
4. 规则驱动当前展示视图，不回写资源数据

凡是不符合这四条的旧代码、旧表义、旧任务，都应该直接删掉或重写，不保留兼容层。
